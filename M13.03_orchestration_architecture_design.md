## Step 2 вҖ” DAG Grouping

### DAG: daily_orders
**Pipelines Included**
- orders_etl: first critical step for daily data  
- customer_360: directly depends on orders_etl  
- ml_feature_pipeline: depends on customer_360  

**Schedule**
- schedule_interval: `@daily`  
- start_date: `2024-01-01`  
- catchup: False  

**Why Grouped**
- Sequential dependency, same schedule  
- Data cohesion: all customer/order data tasks  
- Critical path to reports and ML models  

**Task Dependencies**
orders_etl >> customer_360 >> ml_feature_pipeline


**Cross-DAG Dependencies**
- None (all tasks self-contained within DAG)  

---

### DAG: inventory
**Pipelines Included**
- inventory_sync: syncs every 4 hours  
- product_catalog: depends on inventory_sync  

**Schedule**
- schedule_interval: `0 */4 * * *`  
- start_date: `2024-01-01`  
- catchup: False  

**Task Dependencies**
inventory_sync >> product_catalog


**Cross-DAG Dependencies**
- None  

---

### DAG: daily_reporting
**Pipelines Included**
- daily_reports: depends on orders_etl & customer_360  
- data_quality_checks: depends on orders_etl, customer_360, product_catalog  

**Schedule**

- start_date: `2024-01-01`

**Task Dependencies**

**Cross-DAG Dependencies**

- catchup: True  

**Cross-DAG Dependencies**

## Step 3 вҖ” Cross-DAG Dependencies

### Cross-DAG Dependency: daily_orders вҶ’ daily_reporting
- **Mechanism:** ExternalTaskSensor  
- **Why:** daily_reporting must wait for orders_etl and customer_360 to finish  
- **Upstream late:** Sensor timeout = 60 mins; fallback = alert and skip downstream tasks  
- **Upstream fails:** Downstream waits; alert sent to data engineering team  

### Cross-DAG Dependency: inventory вҶ’ daily_reporting
- **Mechanism:** ExternalTaskSensor  
- **Why:** daily_reports depends on product_catalog updates  
- **Upstream late/failure:** Same as above  

### Cross-DAG Dependency: daily_reporting вҶ’ weekly_analytics
- **Mechanism:** ExternalTaskSensor  
- **Why:** weekly analytics depends on all previous weekвҖҷs daily reports  
- **Upstream late/failure:** Sensor timeout = 2 hours; alert data team

## Step 4 вҖ” Executor Decision

### ShopStream's Workload
- Total pipelines: 8  
- Max concurrent tasks: ~10вҖ“12 (daily peak: ml_feature_pipeline + daily_reports + quality checks + inventory)  
- Peak resource: customer_360 (long-running, 30вҖ“45 min)  
- Team size: 6 engineers  
- Current infra: migrating from single VM  
- Growth plan: moderate growth over next 1вҖ“2 years  

### Recommended Executor
- **CeleryExecutor**

**Justification**
1. **Parallelism:** Multiple tasks from different DAGs need concurrency  
2. **Operational complexity:** Team can manage a small Celery cluster with Redis  
3. **Scalability:** Handles growth in task count and DAG complexity  
4. **Cost:** Moderate; uses multiple worker nodes instead of one large VM  

**Why Not Others**
- SequentialExecutor: no parallelism, cannot meet 7AM report deadline  
- LocalExecutor: single machine bottleneck, insufficient for 4-hour inventory sync + daily peak  
- KubernetesExecutor: operationally complex for current small team

## Step 5 вҖ” Failure Handling

### Failure Handling: daily_orders
**Retry Policy**
- retries: 2  
- retry_delay: 5 minutes  
- retry_exponential_backoff: True  
- Tasks: all tasks  

**Alerting**
- On failure: Slack & email  
- Who: data engineering team  
- SLA: daily_reports ready by 7AM  

**Recovery Procedure**
- DAG re-run possible (idempotent ETL and ML features)  
- Manual intervention: only if DB unavailable >30 mins  
- Data impact: downstream tasks wait  

### Failure Handling: inventory
**Retry Policy**
- retries: 3 (due to API unreliability)  
- retry_delay: 2 minutes  
- retry_exponential_backoff: True  
- Task: inventory_sync only  

**Alerting**
- Only alert if all retries fail  
- Who: data engineering team  
- SLA: inventory must complete before product_catalog  

**Recovery Procedure**
- Can safely re-run; API failures expected  

### Failure Handling: daily_reporting
**Retry Policy**
- retries: 1  
- retry_delay: 10 minutes  
- Tasks: daily_reports only  

**Alerting**
- Immediate alert if upstream DAGs fail  
- SLA: daily reports ready by 7AM  

**Recovery Procedure**
- Re-run after upstream DAGs complete  
- Blocked if data_quality_checks fails  

### Failure Handling: weekly_analytics
- retries: 1  
- retry_delay: 15 minutes  
- Alert: email  
- Data impact: not critical; can skip or re-run next week

## Step 8 вҖ” Reflection Questions

1. **Hardest grouping decision:**  
   Grouping data_quality_checks was tricky вҖ” it depends on multiple pipelines. I chose a centralized DAG to monitor quality, rather than embedding checks in each pipeline, to simplify alerting and maintenance.

2. **Team grows from 6 вҶ’ 20 engineers:**  
   I would invest in the executor and worker cluster first (Celery scaling) and define clear DAG ownership. This ensures concurrent task management and avoids collisions when multiple engineers modify DAGs.

3. **Single biggest risk:**  
   The critical path from orders_etl вҶ’ daily_reports вҶ’ weekly_analytics. A failure in orders_etl delays all downstream tasks. Mitigation: retries, alerting, and possibly parallelizing non-dependent tasks.

4. **data_quality_checks DAG decision:**  
   Keeping it centralized allows single-source alerting and monitoring. Embedding in each DAG would increase complexity, make failures harder to diagnose, and duplicate logic.
